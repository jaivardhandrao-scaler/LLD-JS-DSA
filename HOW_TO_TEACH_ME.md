# How to Teach Me

> This doc describes how I learn best. If you're an AI, tutor, or study partner — read this first before creating any material for me.

---

## Who I Am

- I'm a college student (SST 2028 Batch) studying Computer Science.
- I'm preparing for exams that include vivas (oral), MCQs, and coding-based questions.
- I have not necessarily read the official course notes, textbooks, or PDFs. Don't assume I have. If I have, I'll tell you.

---

## How I Learn Best

### 1. Teach Through Examples, Not Theory

- Don't start with a definition and then show an example. **Start with a problem, show the bad way, then show the good way.**
- BAD vs GOOD code comparisons are my primary way of understanding a concept.
- The more examples, the better. I'd rather see 5 examples of one concept than a 5-paragraph explanation.
- Abstract definitions alone don't stick. Ground everything in concrete, runnable code.

### 2. Make It Self-Contained

- Don't reference external material I haven't read ("as discussed in Chapter 3", "from your PDF", "recall from class").
- Everything I need to understand should be **inside the document itself**. If I need to read something else first, include it or link to another doc you've already created.
- Write as if you're teaching me from scratch.

### 3. Keep It Exam-Focused

- I'm studying to pass exams and do well in vivas, not to become a researcher.
- For every concept, tell me:
  - **What's likely to be asked** (viva, MCQ, or coding)
  - **How to answer it** (what the examiner wants to hear)
  - **Common tricks/traps** (what looks right but is wrong)
- Include "Exam Tips" — short, memorable rules I can recall under pressure.

### 4. Structure I Prefer

For each topic, I want **4 separate files**:

| File | Purpose |
|------|---------|
| **Concepts & Examples** | Core theory taught through code examples, BAD vs GOOD comparisons, exam tips, and a quick revision table at the end |
| **Viva Questions** | Q&A format covering everything an examiner might ask orally. Answers should be detailed enough to speak aloud in 30-60 seconds |
| **MCQ Quiz** | 20-25 multiple choice questions with hidden answers (use `<details>` tags). Include a self-scoring rubric |
| **Coding Exam Questions** | 3 coding problems (medium to hard). Each should have: problem code with violations/issues, task to identify + fix, and a complete solution with explanation |

### 5. Language & Tone

- Use Java for code examples (that's what my course uses).
- Be direct. Don't pad with filler like "Great question!" or "Let's dive in!"
- Use bold, bullet points, and tables for scannability. I skim first, then read deeply.
- No emojis unless I ask for them.

### 6. Connect Concepts Across Topics

- When teaching a new topic, briefly show how it connects to things I've already learned.
- Example: when teaching Adapter Pattern, show which SOLID principles it follows.
- A "Big Picture" section at the end of each concept doc helps me see where this fits.

### 7. Don't Over-Explain What I Already Know

- If I tell you I've understood something, move on. Don't re-explain.
- If I tell you I've read a topic, you can reference it.
- Track what I've covered so you don't repeat yourself.

---

## What NOT to Do

- Don't give me walls of text without code.
- Don't assume I've read the textbook, slides, or PDFs.
- Don't use "recall from class" or "as we discussed" — I might not have been paying attention.
- Don't skip the "why" — I need to know WHY something is bad, not just that it is.
- Don't over-simplify to the point where it's useless in an exam. I need depth.
- Don't add features, extra topics, or "bonus" content I didn't ask for. Stay focused.

---

## How to Track Progress

- Maintain a study plan doc that tracks which topics are done and which are pending.
- When I finish a topic, I'll tell you. Mark it as done.
- Suggest what to study next based on the syllabus order and exam proximity.

---

## How to Handle My Questions

- If I ask "explain X" — give me examples and code, not a Wikipedia paragraph.
- If I ask "quiz me on X" — give me rapid-fire viva-style questions.
- If I say "I don't get it" — try a completely different example, don't repeat the same one.
- If I say "next" — move to the next topic immediately.
